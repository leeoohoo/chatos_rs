### FIX-01 云端失败回执和 Run 详情泄露供应商内部地址

- 发现：云端规划首次因网络波动失败，ChatOS 消息与 Run 详情直接展示 `error sending request for url` 和供应商地址。
- 根因：回执确定性清理只处理内部 ID/枚举，没有处理网络错误与敏感运行时文本；Run 详情也直接渲染历史 `error_message`。
- 修复：后端回执写入与历史读取统一把瞬时网络/5xx 错误转换为中文或英文通用提示；ChatOS Run 详情复用用户可见错误转换器。
- 回归：回执定向测试 13 passed，前端错误转换测试 3 passed，类型检查通过；重启 ChatOS 后真实旧消息和旧 Run 详情只显示“服务暂时不可用，请稍后重试。”，不再出现内部地址。

### FIX-02 “基于当前配置重试”仍固定复用失败 Run 的旧模型

- 发现：云端规划第一次重试仍使用原失败 Run 的模型；切换任务当前模型后，重试接口也不会采用新配置，与界面“基于当前配置重试”不一致。
- 根因：`retry_run_with_user` 显式把历史 Run 的 `model_config_id` 写回新请求，覆盖任务当前默认配置。
- 修复：重试请求不再固定旧模型，由统一启动链读取任务当前 `default_model_config_id`；保留历史 prompt override 和新的 Run/幂等身份。
- 回归：新增策略测试通过；任务先切换到备用模型，真实重试 Run 的 `model_config_id` 随即变为备用配置，证明修复生效。备用 GLM 域名当前无法解析后，再切换到可连通的 `gpt-5.4` 创建新 Run 继续执行。

### FIX-03 规划 Agent 忽略用户明确的任务数量硬约束

- 发现：统一复杂需求明确要求生成 8～14 条可执行项目任务，但云端只生成 6 条、客户端只生成 7 条；任务内容基本覆盖需求，数量约束却被当成建议。
- 根因：规划系统 Prompt 只要求合理粒度和验收覆盖，没有要求在完成前提取并复核用户给出的精确数量、最小值、最大值或范围。
- 修复：规划 Prompt 新增定量约束门禁，要求完成前读取真实产物并逐项核对数量；不足时继续拆分、合并或补齐现有产物，禁止用重复产物凑数。
- 回归：Prompt 针对性测试 1 passed，新版系统 Prompt 已发布；当前项目待实施链结束后执行一次无重复规划修订验证。

### FIX-05 沙箱改动已可自动收集，Agent 仍反复研究逐文件上传

- 回归：Prompt 针对性测试 1 passed，新版系统 Prompt 已发布，后续云端前端和综合验收任务将使用新规则。

### FIX-06 ChatOS 规划 Agent 创建的实施阶段仍被标成“规划任务”

- 发现：云端“实现后端”明确需要编码、测试和沙箱执行，但 Task Runner 页面仍显示“规划任务”，并加载规划任务专用能力集合。
- 根因：子任务无论自身性质如何，都会无条件继承父级 ChatOS Planner 的 `chatos_plan` profile；任务创建协议没有逐项携带 `is_planning_task`。
- 修复：单任务、依赖 DAG 和项目执行任务创建协议均增加显式 `is_planning_task`；规划 Agent 必须逐项填写。旧数据若 profile 为规划但 `requires_execution=true`，运行时和页面按实施任务兼容处理。
- 回归：任务类型与 MCP 能力定向测试 24 passed；Task Runner 前端类型检查和生产构建通过；真实既有“实现后端”任务已从“规划任务”显示为“普通任务”。

### FIX-08 过期项目运行拓扑会在模型获得工具前阻断实施任务

- 发现：云端补齐任务开始时，项目默认分支已经升级为 React/Vite，但项目环境仍保存旧后端时期的 Dockerfile；构建执行 `npm run build` 时缺少当前前端依赖，Run 在模型执行前直接失败。
- 根因：Task Runner 对单应用项目自动使用项目运行拓扑，拓扑 Dockerfile 可能落后于当前默认分支；环境构建失败时没有基础执行沙箱降级路径，导致任务无法进入终端修复构建本身。
- 修复：环境拓扑继续优先启动；仅当任务未显式指定应用服务、且存在已就绪同运行时镜像时，环境启动失败后自动切换到基础执行沙箱。显式服务选择不降级，避免悄悄丢失依赖拓扑。
- 回归：路由定向测试 7 passed；真实 Run `b334850a-baef-482f-8edb-82c9289e9274` 先记录项目环境失败，再使用 `dev-node24` 成功获得基础沙箱和终端工具。

### FIX-09 已发布工程执行 Prompt 正文与 checksum 不一致

- 发现：基础沙箱成功后，Run 在解析 `task_runner_run_phase` 时返回 `agent_prompt_checksum_invalid`。
- 根因：数据库正文保存时去掉了末尾换行，但历史发布 checksum 按原始文件计算，导致运行时完整性校验必然失败。
- 修复：使用系统种子同步逻辑重新发布所有 vendor 的标准 Prompt，并统一按最终持久化正文计算 checksum；revision 从 4 更新为 5。
- 回归：Prompt 种子测试 5 passed；内部 gpt Prompt 解析返回 HTTP 200，正文长度 1127，实际 SHA-256 与持久化 checksum 一致；后续 Run 已进入模型和工具循环。

### FIX-10 客户端项目根路径会把绝对路径静默改成相对路径

- 发现：本地项目创建接口原先对 `root_relative_path` 直接去除首尾 `/`，例如绝对路径会变成看似合法的相对路径，边界语义错误。
- 根因：项目创建路径没有复用工作区相对路径校验，绝对路径在解析前被破坏。
- 修复：显式拒绝绝对路径，并复用统一相对路径规范化和 canonical 边界校验。
- 回归：单元测试通过；新版 Release Core 真实发布到 39234。绝对路径、`..`、不存在目录、指向工作区外的符号链接均返回 `local_runtime_project_root_invalid` 和 HTTP 400，合法相对目录创建成功后已清理测试记录。

- 修复：Activity 契约增加中文 `summary`，后端按资源和动作生成中文摘要，数据导入隐藏内部标题；前端仅渲染摘要，并中文化状态、优先级、频率、页头和错误边界。
- 回归：API 10/10、严格类型检查和生产构建通过；真实浏览器首页、目标、习惯、活动页均显示中文，390 和 1280 视口 `scrollWidth == clientWidth`。

### FIX-12 Prompt 初始化失败回执泄露内部模块、vendor 和 checksum

- 发现：Prompt checksum 故障的失败回执直接展示 `task_runner_run_phase`、`vendor gpt`、HTTP 409 和内部校验枚举。
- 根因：用户可见错误转换覆盖网络、5xx、令牌和部分内部 ID，但没有覆盖平台初始化、Prompt 解析、沙箱拓扑和 worker 内部失败。
- 修复：统一回执清理新增平台内部失败分类；中文显示“任务暂时无法启动，请稍后重试。”，英文显示等价通用提示。新回执写入和历史/Turn/compact 读取共用同一转换。
- 回归：新回执与历史读取定向测试均通过；重启 ChatOS 后，真实旧失败回执的正文、结果摘要、错误和详情预览均只返回中文通用提示。

### FIX-13 项目联系人区域泄露内部 Agent ID

- 发现：项目“用户消息”页面的联系人卡片，以及更换联系人弹窗，会把内部 Agent ID 直接展示给普通用户；联系人缺少名称时还会把内部 ID 当作显示名称。

- 修复：运行数据继续保留内部关联字段，界面只显示联系人名称；缺少名称时使用本地化的“未命名联系人”，选择弹窗不再接收或渲染 Agent ID。
- 回归：新增前端定向测试，类型检查、Lint 和生产构建通过；真实 ChatOS 项目页面只显示联系人“叽咕狸”，DOM 中不再出现 Agent ID 或其值。

### FIX-14 项目任务重试成功后需求仍永久显示失败

- 发现：首个云端实施任务因环境和 Prompt 初始化问题失败，修复后同一任务重试成功，但需求卡仍保持“失败”，即使后续任务已经运行。
- 根因：项目任务状态可以从失败恢复为进行中或完成，需求聚合状态机却只允许从已确认/进行中进入完成，没有在失败或阻塞原因消失后重新计算需求状态。
- 修复：每次项目任务恢复为待处理、就绪、进行中或完成时，重新聚合当前需求及父需求子树；仍有失败则保留失败，仍有阻塞则显示阻塞，全部完成则完成，否则恢复为进行中。取消和归档不会误触发恢复。
- 回归：MongoDB 集成状态机 7 项全部通过，新增“失败后重试恢复进行中”和“失败后重试成功完成”用例；重启 Project Management 后，以真实运行中的第二阶段对账，需求页面从“失败”恢复为“进行中”。

### FIX-15 成功回执把完整技术报告直接塞进聊天

- 发现：第二阶段成功后，ChatOS 回执整段展示文件名、API 路径、数据库术语、测试命令和原始错误枚举，普通用户需要从大段工程报告中寻找结论。
- 根因：成功回执优先使用 Task Runner 的完整 `result_summary/report_content`，只做敏感字段替换，不区分聊天回执与任务详情的展示层级。
- 修复：成功聊天回执改为确定性的简短结果：“本阶段已完成并通过任务内验证。完整技术报告可在任务详情中查看。”；完整工程报告继续保留在任务详情。历史回执读取也执行同一收口，因此旧消息会自动变得简洁。失败回执仍保留经过清理的可操作错误。
- 回归：ChatOS 回执与历史处理定向测试 21 项通过；重启后真实第二阶段旧回执已自动折叠，页面不再包含文件名、API 路径、内部数据库字段或原始错误枚举。

### FIX-16 沙箱产物清单把依赖缓存误报成业务改动

- 发现：第二阶段实际只修改 7 个源码/测试文件，沙箱产物清单却把 `node_modules`、Vite 缓存等 5600 多个文件统计为新增，拖慢输出收集并让 Run 详情严重失真。
- 根因：Sandbox Manager 导出结果时只排除了平台内部目录，没有排除语言依赖和构建缓存；Harness 最终提交虽会遵守 Git 忽略规则，但此前的复制、哈希、差异和预览已经产生了大量无效工作。
- 修复：输出工作区和变更清单统一跳过 Node、Python、Rust 的常见依赖/缓存目录，包括 `node_modules`、包管理器缓存、Vite 缓存、虚拟环境、Python 缓存和 Rust `target`；真实源码和发布文件仍正常收集。
- 回归：Sandbox Manager 全量 27 项通过、1 项 Docker 专项按环境忽略；热更新后第三阶段真实沙箱清单从前一阶段的 5613 项降为 13 项，完整保留 9 个修改和 4 个新增业务文件，随后 Harness 提交成功。

### FIX-17 纯规划任务的固定工具集仍包含终端

- 发现：Task Runner 全量测试有两条旧断言要求 `chatos_plan` 任务固定获得终端；这与“纯规划不得强制沙箱、不得修改项目文件”的当前契约冲突，也会让只读 Harness 规划在运行时携带不必要的本机终端能力。
- 根因：任务性质已通过 `requires_execution` 区分纯规划和历史误标实施任务，但纯规划固定能力表仍保留旧的 `TerminalController`。

### FIX-18 浏览器依赖和大型临时文件会耗尽受限目录

- 发现：云端真实 Chromium 验收需要安装浏览器和 59 个动态库，默认临时目录容量不足时会中断下载或解压。
- 根因：工程执行 Prompt 没有明确区分小型临时脚本与大型浏览器、依赖缓存；Agent 容易把大型文件放入容量受限的 `/tmp` 或用户主目录。
- 修复：Task Runner 工程执行 Prompt 要求大型临时文件统一放到项目工作区 `.chatos/tmp`，小型短生命周期脚本才可使用系统临时目录；依赖仍不得写入项目清单。
- 回归：新版 Prompt 已发布；最终云端验收在工作区保存浏览器运行依赖和证据，完整通过双视口 86 项检查。

### FIX-19 沙箱进程上限与僵尸浏览器进程

- 发现：Chromium 多进程模型曾触达 128 PID 上限；提高到 512 后又发现容器 PID 1 不回收孤儿 Chromium，留下多批 zombie，长期运行仍可能耗尽 PID。
- 根因：默认 `max_processes` 太低，且沙箱容器直接以 MCP Server 或运行启动脚本作为 PID 1，没有轻量 init 负责信号转发和孤儿回收。
- 修复：默认进程上限提高到 512；Docker CLI 沙箱、项目应用容器和 Docker API 沙箱统一启用 `--init`/`HostConfig.Init=true`。
- 回归：Sandbox Manager 29 passed、1 个 Docker 专项按环境忽略；服务重启后新建真实 Node.js 24 沙箱，Docker 显示 PID 1 为 `docker-init`、PIDsLimit=512，制造 40 个孤儿子进程后 zombie 数为 0，随后租约正常释放。
