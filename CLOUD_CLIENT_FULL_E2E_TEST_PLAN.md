# Chat OS 云端与 macOS 客户端全流程测试计划

## 1. 测试目标

以当前工作区源码、当前本地开发云服务和最新 macOS Local Connector 客户端为准，重新完成一次端到端验收。测试不只验证页面能打开，还要验证真实数据落库、模型调用、项目规划、任务执行、沙箱路由、成功/失败/停止/重试回执、客户端重启恢复和最终安装包。

执行原则：

- 先列出测试点，再按编号逐项执行并即时更新结果。
- 遇到产品问题直接定位、修复并补针对性回归，不只记录原因。
- 不清理或覆盖用户已有项目和未提交代码；测试数据使用可识别的 E2E 名称。
- 不在文档、日志或回执中记录密码、API Key、访问令牌等敏感信息。
- 云端和客户端产生的需求、文档、任务与回执必须跟随用户的中文语言。

## 5. 问题与修复记录

### FIX-01 新增 API 未同步基线且前端热点文件超限

- 发现测试点：A02。
- 现象：API surface/path 各增加 1 条后基线检查失败；`useChatStreamRealtimeBridge.ts` 达到 433 行，超过 412 行限制。
- 根因：新增项目镜像生成接口后漏更新仓库基线；回执对账职责继续堆入已有热点 Hook。
- 修复：同步两份 API 基线；抽出独立的当前会话回执对账 Hook，主文件降至 411 行。
- 回归证据：API surface、API path、热点行数和大文件检查全部通过。

### FIX-02 Task Runner 回执直接展示内部 ID 和原始枚举

- 发现测试点：D04、F04。
- 现象：真实成功回执向普通用户展示 `requirement_id`、`document_id`、`work_item_id`、UUID、`implementation_plan` 和 `blocks`。
- 根因：回执包装层把模型的完整 `result_summary` 原样拼入用户消息，Prompt 约束无法提供确定性保护。
- 修复：回执内容在服务端进行确定性用户视图清理；内部 ID 行和 UUID 不进入正文，常见文档/依赖枚举转换为中文用户术语，原始诊断数据仍保留在内部记录。
- 回归证据：回执测试 11 passed，新增用例确认正文和 `detail_preview` 均不包含内部 ID、UUID 和原始枚举；等待下一条真实回执复核。

### FIX-03 工作节点心跳过期只改状态、不投递失败回执

- 发现测试点：F05、F09、F11。
- 现象：服务整体重启后，运行中的任务因旧 Worker claim 过期被标记失败，但 ChatOS 没有任何失败回执。
- 根因：过期 claim 回收直接在存储层批量更新 Run，绕过统一终态回调逻辑，也没有同步追加可观察的 Run 事件。
- 修复：存储层返回实际被回收的 Run；服务层同步任务状态、追加 `run.claim.expired` 事件，并调用统一终态回调投递。
- 回归证据：存储层定向测试通过；已注入真实过期运行，等待 Worker 回收和 ChatOS 自动回执完成集成验证。

### FIX-04 页面刷新后没有恢复当前项目上下文

- 发现测试点：B03。
- 现象：登录状态、项目列表和会话数据可以恢复，但刷新后回到公共联系人会话，没有回到刷新前选中的云端项目。
- 根因：首次启动时项目列表与会话列表并发加载，会话自动选择先落到公共范围，覆盖稍后恢复的最近项目。
- 修复：启动顺序调整为先恢复项目，再加载并选择对应项目会话；补充延迟项目加载回归用例。
- 回归证据：浏览器真实刷新后仍停留在 `cloud-client-e2e-20260719-2130` 的原会话，历史消息与 Plan 上下文完整。

### FIX-05 空云端项目目录显示原始 Harness 404

- 发现测试点：C01、F12。
- 现象：刚创建的空项目目录同时显示“目录为空”和原始 `revision \"main\" not found` 404 JSON。
- 根因：Harness 空仓库尚无初始默认分支，根目录读取返回 `revision main not found`，Project Management 将其当作普通目录错误。
- 修复：仅对空仓库根目录缺少初始修订的情况返回空目录；普通缺失路径仍保留 404。
- 回归证据：Project Management 定向测试通过；浏览器真实打开空项目目录只显示“目录为空”，不再显示原始 404。

### FIX-06 Task Runner 回执只有进程内短暂重试

- 发现测试点：F09、F11。
- 现象：ChatOS 暂时不可达时仅执行 0/250/750ms 三次尝试，之后回执可能永久丢失；服务重启也无法恢复。
- 根因：运行记录没有持久投递状态，后台没有待补投扫描器。
- 修复：终态 Run 原子携带 `pending/delivered/skipped` 投递状态、尝试次数、下一次补投时间和最后错误；新增后台补投器、最大 60 秒退避和启动时立即补投。
- 回归证据：真实停止 ChatOS 后端后，Run 成功但回执保持 `pending` 并累计失败；恢复 ChatOS 并重启 Task Runner 后自动转为 `delivered`，ChatOS 页面无需手工刷新即出现消息。

### FIX-07 联系人异步失败任务无法重试

- 发现测试点：F07。
- 现象：Task Runner 页面提供“重试”，后端却返回 `contact_async tasks can only be started by the scheduler`。
- 根因：显式重试复用了首次手工启动限制，没有区分首次启动与失败后的重试。
- 修复：新增 Retry 触发来源；联系人异步任务仍禁止普通手工首启，但允许调度器启动和终态 Run 的显式重试。
- 回归证据：原失败 Run `5b87cbd1-1436-4663-b86d-b5f872c423aa` 成功创建新 Run `3980e2a8-716f-45fd-b287-98ca525ca914`，规划执行成功且未启用沙箱。

### FIX-08 回执仍会泄露非标准内部标识和运行时术语

- 发现测试点：D04、F12。
- 现象：模型返回非标准长度内部标识、`ready=true`、内部依赖检查函数名、原始需求状态和英文文档枚举时，旧 UUID 过滤无法识别。
- 根因：清理逻辑只匹配标准 36 字符 UUID，且只转换少数下划线枚举。
- 修复：支持剥离 6-8/4/4/4/10-12 格式的内部标识；按消息语言转换依赖、环境状态、需求状态和文档术语；移除多余代码样式。
- 回归证据：回执测试 12 passed；同一真实回执幂等重投后页面仍只有 1 条消息，正文不再包含内部标识、`ready=true`、内部函数名、`pending` 或英文 `implementation plan`。

### FIX-09 未初始化默认镜像可被选择并导致创建长期卡住

- 发现测试点：G01、G04、G08。
- 现象：默认镜像 `chatos-sandbox-agent:latest` 实际未初始化，但创建页仍将其作为默认可选项；点击后后端直接执行 `docker run` 并尝试拉取镜像，页面长期停留在“创建中”。服务重启时该 Docker CLI 子进程还可能短暂成为孤儿进程，并占用沙箱池活跃名额。
- 根因：前端只禁用“可构建且未初始化”的镜像，没有禁用未初始化的配置默认镜像；表单初始化早于镜像目录返回，未自动切换到第一个就绪镜像；后端对 `default` 分支绕过了镜像存在性检查；本地开发栈停止服务时只根据父进程存活状态决定是否强杀进程组。
- 修复：创建页对所有未初始化镜像禁用并标注原因，镜像目录加载后自动选择配置默认镜像或第一个就绪镜像，没有就绪镜像时禁用创建按钮；后端所有镜像（含 `default`）在创建前统一检查本地就绪状态；本地开发栈停止逻辑独立检查并清理完整进程组。
- 回归证据：JDK 21 自动成为创建页默认镜像，未初始化 Default 选项为 disabled；直接请求 Default 在 0.03 秒内返回 HTTP 400 明确错误且不启动 Docker；后端镜像测试 3 passed、前端类型检查通过；进程组回收回归确认忽略 TERM 的子进程仍会被完整清理，沙箱池恢复 `active=0/pending=0`。

### FIX-10 旧 Task Runner 回执历史仍暴露内部实现内容

- 发现测试点：D04、F12。
- 现象：新回执已清理，但刷新历史后旧回执仍可能展示 requirement/document/work item 标识、UUID、`draft`、`pending`、`implementation_plan`、`blocks`、`contains` 和内部依赖函数名。
- 根因：确定性清理只发生在新回调写入阶段，历史读取和旧元数据摘要没有经过同一用户视图转换。
- 修复：抽出统一回执显示清理模块；新回执写入、历史消息读取和旧元数据摘要全部复用同一规则。
- 回归证据：回执测试 12 passed、历史处理测试 10 passed；真实浏览器刷新后旧回执不再显示内部 ID、原始枚举和内部函数名。

### FIX-11 停止接口没有真正取消上游模型 HTTP 请求

- 发现测试点：C06。
- 现象：点击停止只改变本地 aborted 标记，上游模型请求仍继续等待，下一轮可能被旧请求占用。
- 根因：轮次 `CancellationToken` 没有贯通到 AI Runtime 的 HTTP/流式请求。
- 修复：每个轮次创建独立取消令牌，按会话和轮次传入 AI Runtime；停止接口只取消匹配的活动轮次，下一轮会替换旧令牌。
- 回归证据：Abort registry 6 passed、Chat execution 15 passed；真实慢模型请求在点击停止后后端记录 `error="aborted"`，测试端点检测到客户端连接断开。

### FIX-12 迟到实时事件会错误收口新轮次，刷新后也无法恢复停止按钮

- 发现测试点：C06、F11。
- 现象：只持久化用户消息、旧轮次助手终态或旧轮次失败事件都可能清空新轮次的 `isLoading/activeTurnId`；页面刷新后正在运行的普通聊天也失去停止按钮。
- 根因：实时桥接没有按活动 turn 做终态匹配，且会话选择只恢复历史消息，没有读取后端运行快照的 `active_in_runtime`。
- 修复：只有同一活动 turn 的助手终态、失败或取消事件才能收口当前状态；会话恢复时读取最新运行快照，真实活动轮次恢复 `activeTurnId` 和停止按钮，迟到旧快照不得覆盖新轮次。
- 回归证据：前端相关测试 25 passed、类型检查通过；真实慢请求运行中刷新页面后停止按钮仍存在，点击后消息变为“已取消”，随后新消息立即得到 `CANCEL_CONTINUE_OK`。

### FIX-13 模型错误把密钥、内部 trace 和协议正文直接展示给用户

- 发现测试点：C07。
- 现象：上游 HTTP 500 的原始 JSON 会在 ChatOS 横幅中显示，其中包含 `api_key`、`internal_trace`、HTTP 状态和解析协议细节；取消也显示英文 `Chat turn cancelled` 错误横幅。
- 根因：未分类错误直接作为实时事件 `message/detail` 下发，前端又把原文写入全局错误和失败消息。
- 修复：后端用户错误事件不再下发原始 detail，对认证、限流、5xx、网络和解析异常输出稳定中文错误码与文案；前端增加第二层敏感字段和协议文本过滤；用户主动取消不生成错误消息或错误横幅。
- 回归证据：后端错误事件测试 5 passed、前端错误与实时桥接测试通过；真实 500 回归只显示“模型服务暂时不可用，请稍后重试或切换模型。”，页面不包含测试密钥或内部 trace。

### FIX-14 服务重启后的孤儿轮次永久显示“正在处理”

- 发现测试点：C05、C06、F11。
- 现象：后端重启中断的普通聊天只有用户消息，持久快照长期保持 `running`，历史页面永久显示“正在处理”。
- 根因：历史显示只信任用户消息中的 `overall_status`，没有结合持久运行快照和当前进程真实活动轮次进行对账。
- 修复：紧凑历史读取时，对无实际 Task Runner 子任务的联系人异步消息执行运行态对账；当前进程真实活动的轮次保持“正在处理”，快照仍为 running 但当前进程不活动的孤儿轮次显示为“已取消”。
- 回归证据：历史处理测试 14 passed；真实加载更多历史后，23:57 和 00:01 两条遗留消息由“正在处理”自动变为“已取消”，活动慢请求仍保持可停止状态。

### FIX-15 系统 Agent Prompt 源码升级后运行时仍加载旧版本

- 发现测试点：H02。
- 现象：四个语言敏感系统 Prompt 已在源码加入“跟随用户语言、保留真实代码标识符”等规则，但数据库和运行时代理仍返回旧版本；其中 GLM 主对话 Prompt 仍是缺少语言规则的英文版本。
- 根因：系统 Prompt 播种只在记录不存在时写入，已有 `system_seed` 记录永远不会跟随版本升级；旧管理接口编辑系统 Prompt 时也没有把来源转换为管理员自定义，升级逻辑无法区分旧遗留记录。
- 修复：系统 Prompt 保存基线校验值；未被管理员修改的系统 Prompt 在启动时自动升级、增加修订并生成版本记录；管理员开始编辑时立即转换为自定义来源，后续升级不覆盖；旧 `system_seed` 记录执行一次兼容迁移。
- 回归证据：播种策略测试 5 passed；Plugin Management 重新启动后 bundle 升级到版本 5，经 Local Connector Service 真实代理读取 44 条 Prompt，四个语言敏感 Agent 的 16 个供应商版本全部包含用户语言与代码标识规则。

### FIX-16 配置中心离线回退没有把当前快照标记为过期

- 发现测试点：H05。
- 现象：配置中心不可达时 SDK 返回的快照带有 `stale=true`，但内存中的当前快照仍保持新鲜状态；后续心跳、策略同步和状态页面可能误报配置未过期。
- 根因：内存与磁盘回退分支只修改返回值，没有把带来源和过期标记的快照重新安装为当前快照。
- 修复：内存回退和本地缓存回退均先写入 `source` 与 `stale=true`，再作为当前快照安装；配置中心恢复并返回 304 时继续按既有逻辑恢复为新鲜状态。
- 回归证据：Config SDK 3 passed，新增用例分别验证磁盘缓存和内存快照在配置中心不可达时返回并持久保持正确的过期来源；当前真实服务实例均读取配置修订 4，Memory Policy 代理显示 `stale=false`、来源为配置中心。

### FIX-17 历史孤儿轮次会永久锁住项目联系人

- 发现测试点：B06、C08。
- 现象：Task Runner、普通聊天和运行快照均已终态，项目页仍提示“当前项目有运行中的任务”，更换与解绑联系人按钮持续禁用。
- 根因：联系人锁定接口直接读取历史消息里遗留的 `processing` 标记，没有复用聊天历史已经具备的运行快照与当前活动轮次对账逻辑。
- 修复：联系人锁定扫描在判断历史轮次前，先结合持久运行快照和当前进程活动轮次修正联系人异步状态；真实活动轮次仍保持锁定，服务重启遗留的孤儿轮次不再锁定。
- 回归证据：联系人锁定定向测试 4 passed；ChatOS 后端重启后真实项目页面不再显示错误锁定提示，“更换”和“解绑”按钮恢复可用。
