# Chat OS 云端与客户端统一复杂验收需求

## 1. 用途

本文件定义一份长期可复用的复杂 E2E 验收需求。每次完整回归时，分别在 Chat OS 云端和 macOS 客户端创建独立项目，输入完全相同的需求，验证从首次建项、联系人和模型选择，到中文规划、任务执行、代码生成、自动化测试、真实运行、失败恢复和最终回执的完整链路。

本需求不是示例文案。测试产生的需求、技术文档、任务、代码、数据库、测试结果和执行回执都必须是真实可检查的产物。

## 2. 测试项目命名

- 云端项目：`FocusFlow-cloud-e2e-<日期时间>`
- 客户端本地项目：`FocusFlow-local-e2e-<日期时间>`
- 产品名称：`FocusFlow 目标与习惯管理工作台`
- 默认语言：简体中文
- 两个项目必须使用独立目录、独立数据库和独立运行端口，不共享生成产物。

## 3. 统一用户需求原文

> 请为我规划并实现一个名为“FocusFlow”的目标与习惯管理工作台，包含前端、后端和本地持久化数据库。它面向希望把长期目标拆成里程碑、用每日习惯推动目标完成，并能回顾进展的个人用户。
>
> 用户可以创建、编辑、归档和删除目标；每个目标包含标题、说明、状态、优先级、开始日期、目标日期、进度和多个里程碑。用户可以创建每日或每周习惯，把习惯关联到目标，并按日期完成或撤销打卡。同一习惯同一天重复打卡必须保持幂等。首页需要展示今日待办、连续打卡天数、近 7 天完成率、即将到期目标和最近活动。
>
> 系统需要提供目标、习惯和活动记录的搜索、状态筛选、标签筛选与排序；提供目标详情、习惯详情和统一活动时间线；所有新增、修改、归档、删除和打卡操作都写入审计记录。编辑目标和习惯时使用版本号做乐观并发保护，旧版本提交应返回 409，并保留已有数据。
>
> 提供 JSON 数据导出和导入预检。导入预检只返回新增、更新、跳过和错误数量，不写数据库；确认导入后才落库。非法数据必须返回清晰的字段错误，不能导致服务崩溃。提供健康检查接口，并在数据库重启后保留数据。
>
> 前端使用 React、TypeScript 和 Vite，后端使用 Node.js ESM REST API，数据库使用 Node.js 内置 SQLite。不得依赖外部 SaaS。界面必须支持桌面和 390×844 移动视口，包含加载、空数据、错误、重试、确认删除和 404 状态；键盘可操作，表单字段有可识别标签。最终要提供 README、环境变量示例、初始化脚本、开发/测试/构建命令和可复现的示例数据。
>
> 请先生成中文需求、技术方案、实施任务和依赖关系，再逐项执行实现。规划任务不得修改项目文件，也不得强制申请沙箱；实施任务必须在授权项目目录内完成。每项任务完成或失败后都要给出中文、用户可理解的回执，不展示内部 ID、原始枚举、工具协议或敏感信息。

## 4. 固定业务规则

### 4.1 目标与里程碑

- 目标状态仅允许：未开始、进行中、已完成、已归档。
- 优先级仅允许：低、中、高。
- 目标进度为 0～100 的整数；目标标记完成时进度自动为 100。
- 里程碑属于唯一目标，支持标题、截止日期和完成状态。
- 删除目标前必须二次确认；有关联习惯时返回冲突说明，不静默级联删除。

### 4.2 习惯与打卡

- 频率支持每日，以及指定星期的一周多次。
- 每个习惯可以关联 0 或 1 个目标。
- 同一习惯与同一自然日最多一条有效打卡记录。
- 重复完成打卡返回同一业务结果；撤销不存在的打卡也应保持安全幂等。
- 连续天数和近 7 天完成率必须由真实打卡数据计算，不写死在前端。

### 4.3 搜索、筛选与活动

- 目标与习惯支持关键词、状态、标签和排序组合查询。
- 标签比较不区分大小写，返回值保留首次创建时的显示形式。
- 活动时间线支持分页，并显示用户可理解的中文事件摘要。
- 审计数据不得包含访问令牌、API Key、密码或完整请求头。

### 4.4 并发、导入与错误处理

- 可编辑资源包含从 1 开始的 `version`；成功修改后递增。
- 旧 `version` 更新返回 HTTP 409 和当前版本，不覆盖新数据。
- 导入预检与确认导入共用同一校验规则。
- 无效日期、空标题、未知枚举、越界进度和重复业务键返回 HTTP 400/409。
- 任一 4xx 请求后健康检查仍必须返回 200。

## 5. 固定技术约束

- 前端：React + TypeScript + Vite。
- 后端：Node.js ESM + REST API。
- 数据库：Node.js 内置 SQLite；启动时自动建表和迁移。
- 代码结构至少包含 `frontend`、`backend`、共享类型或契约目录。
- 后端监听地址和数据库路径由环境变量配置。
- 根目录提供一键 `test`、`build` 和开发启动命令。
- 自动化测试不得依赖互联网、真实时间等待或测试顺序。
- 生成与运行只能发生在当前项目授权目录内。

## 6. 必须生成的规划产物

- 1 条完整中文需求，包含范围、业务价值、非目标和验收标准。
- 至少 2 份中文技术文档：总体设计；数据/API 与测试发布方案。
- 8～14 条可执行任务，区分规划任务与实施任务。
- 清晰的无环依赖 DAG，至少覆盖：工程基础、数据层、API、前端、联调、测试、真实验收。
- 规划结果不得包含“本轮不要改文件”等临时操作约束。
- 重复规划时应复用或明确修订已有产物，不得静默制造重复需求和任务。

## 7. 必须通过的自动化验收

### 7.1 后端

- 健康检查、初始化示例数据和数据库重启持久化。
- 目标、里程碑、习惯、打卡和活动时间线 CRUD/查询。
- 打卡完成与撤销幂等。
- 关键词、状态、标签、日期和排序组合筛选。
- 乐观并发成功更新与 409 冲突。
- 导出、导入预检和确认导入。
- 400、404、409 后服务仍健康。

### 7.2 前端

- 首页统计来自 API，而不是固定常量。
- 创建目标、编辑目标、创建习惯、完成与撤销打卡。
- 搜索、筛选、清除筛选和空状态。
- 加载、错误、重试、删除确认、冲突提示和 404。
- 桌面导航与移动端折叠菜单。
- 关键表单具备可访问名称并支持键盘提交。
- 中文界面的 HTML 文档语言必须为 `zh-CN`，不能只把可见文本翻译成中文。

### 7.3 构建与边界

- 根目录测试全部通过，前端严格类型检查和生产构建通过。
- 客户端终端默认目录是项目根目录。
- `..`、绝对路径、未授权目录和符号链接越界访问被拒绝。
- 任务停止后不残留占用端口的 Node/npm 子进程或沙箱租约。

## 8. 必须通过的真实交互验收

使用真实浏览器和真实后端完成以下数据链：

1. 创建目标“发布 FocusFlow 1.0”，优先级高，添加 2 个里程碑和标签 `Release`。
2. 创建每日习惯“完成 30 分钟核心开发”，关联该目标。
3. 当日完成打卡两次，确认只产生一条有效记录；撤销后再次完成。
4. 编辑目标进度为 60%；用旧版本再次提交，确认收到冲突且进度仍为 60%。
5. 使用关键词、状态和标签组合筛选并清除筛选。
6. 查看首页近 7 天完成率、连续天数、即将到期目标和活动时间线。
7. 导出 JSON；对一份含新增、更新和非法项的数据执行预检，确认未落库；修正后确认导入。
8. 刷新页面并重启后端，确认目标、习惯、打卡和活动仍存在。
9. 验证不存在资源 404、非法进度 400、旧版本 409，随后健康检查仍为 200。
10. 在 1280 宽桌面和 390×844 移动视口完成导航，无横向溢出和不可操作区域。

## 9. Chat OS 全链路验收点

- 新项目第一次添加联系人、选择模型后，第一条需求立即发送，模型选择不丢失。
- 第二条消息不重新选择模型也可发送。
- 需求、文档、任务和最终回执均跟随中文。
- 纯规划任务不创建沙箱租约、不修改文件。
- 规划完成后必须从项目管理的“执行关联任务”链路启动实施；每个项目任务都要绑定实际 Task Runner 任务并由回调更新状态，不得另建一组未绑定的实施 DAG 造成代码已交付但 Plan 仍显示待办。
- 实施任务按项目运行配置选择云端沙箱或客户端本地运行。
- 成功、失败、停止、重试均产生可恢复且不重复的 ChatOS 回执。
- 服务或客户端短暂中断后，登录态、项目、会话、模型、Plan 和 Run 能自动恢复或对账。
- 最大迭代使用统一 600 轮，并在收口阶段停止调用工具、生成真实总结。
- 用户界面不展示内部 ID、原始枚举、内部地址、工具协议、密钥或长堆栈。
- 最终收口时项目管理必须显示 0 个未完成项目任务；补测或修复产生的任务也必须完成、取消或归档，不能遗留历史假待办。
