# 过程消息与工具执行时间线整体修复计划

## 目标

统一修复本地会话和云端会话的过程消息数量、加载、筛选与展示问题，并把工具调用改成面向用户的动作摘要：不展示内部工具名和调用 ID，只说明“读取了什么、搜索了什么、执行了什么、修改了什么”，点击展开后查看执行结果或错误。

## 已确认问题

1. 本地会话的过程消息真实存在，但本地适配层返回的消息没有 `historyProcessLoaded` 等云端标记，弹窗把所有消息过滤掉。
2. 云端左侧数量来自 Memory Engine 的 turn 统计，弹窗内容来自会话消息查询；当过程记录被隐藏、压缩或只剩合成消息时，两边会出现“8 条但只展示 1 条”的差异。
3. 时间线只读取部分工具调用字段，并把一个消息中的多个 thinking/text 段合并成一个事件，导致事件数进一步减少。
4. 当前工具卡暴露工具名、调用 ID 和大段参数摘要，不符合用户希望的简洁动作表达。
5. 任务运行详情仍使用旧的运行事件时间线，直接展示生命周期事件和 payload，没有复用会话“执行过程”的动作卡片与展开交互。

## 实施步骤

- [x] 云端 turn 消息接口优先从 Memory Engine 的 process-records 恢复完整过程记录，并在失败时回退到现有会话消息逻辑。
- [x] 本地与云端前端统一按“当前用户消息 + 最终助手消息”的边界识别过程消息，同时兼容现有 history process 标记。
- [x] 工具调用读取同时兼容 metadata 与消息顶层的 `toolCalls/tool_calls`，并按调用 ID 去重、关联工具结果。
- [x] 将一个过程消息中的多个 thinking/text 段拆成独立时间线事件，避免多段内容被合并成一条。
- [x] 新增工具动作摘要模块，按读取、搜索、执行、修改、浏览器、网页、远端、任务、记事本、记忆/Agent 等类型提取主要参数。
- [x] 工具时间线默认只显示动作摘要和状态，不显示工具名、调用 ID；展开后显示主要参数、完整结果或错误。
- [x] 弹窗标题同时显示“过程消息数”和“实际解析事件数”，加载完成后以实际返回内容为准，避免误导。
- [x] 增加本地无标记消息、云端恢复消息、多工具调用、多 thinking 段、工具摘要与展开结果测试。
- [x] 新增 Task Runner 运行事件适配层，把 `tools_start` 中的每个调用拆成独立动作，并按 `tool_call_id` 将 `tool_stream` 最终结果合并回对应卡片。
- [x] 将任务“运行详情”的主区域改为与会话过程弹窗一致的四项统计、时间线节点和动作卡片；思考、模型输出与模型请求使用简洁过程卡片展示。
- [x] 运行详情不再默认展示原始 payload；原始运行事件仅保留在底部“原始运行事件（诊断）”折叠区，默认关闭。
- [x] 保留运行事件分页能力，加载更多后重新计算过程步骤、工具完成状态和统计数字。
- [x] 运行后端相关测试、前端全量测试、TypeScript、ESLint 和生产构建；全量并发中的原有 Rust 错误码导出测试曾触发 60 秒钩子超时，隔离重跑 8/8 通过。
- [x] 运行详情同步改造完成后再次执行 ChatOS 前端全量测试 121 个文件 / 422 项、TypeScript 类型检查、相关文件 ESLint 与生产构建，全部通过。
- [x] 部署 Chat OS 前端到本机客户端，并单独重启本地开发环境的 ChatOS 后端；客户端重新加载后即可验证。

## 展示口径

- 读取文件：`已读取 model.ts`
- 读取目录：`已读取 projectPlanPane 目录`
- 搜索文本：`已在 projectPlanPane 中搜索「completed」`
- 执行命令：`已执行 npm test -- --run`
- 修改文件：`已修改 model.ts`
- 应用补丁：`已修改 3 个文件`
- 打开网页：`已打开 https://example.com`
- 网页搜索：`已搜索网页「关键词」`
- 远端命令：`已在服务器名称执行 command`
- 任务操作：`已创建任务「标题」`、`已完成任务「标题」`

失败时使用相同动作的失败态，例如 `读取 model.ts 失败`；运行中使用 `正在读取 model.ts`。工具原始名称仅保留在数据内部，不在普通界面展示。

## 验收标准

1. 本地截图中的 16 条过程消息能展开看到模型过程、工具调用与工具返回。
2. 云端显示的过程数量与实际加载记录一致；若历史记录无法恢复，界面明确显示恢复后的实际数量而不是空白或虚高数字。
3. 一个消息包含多个工具调用或多个 thinking 段时，每个操作都形成独立事件。
4. 默认列表只显示用户能理解的动作摘要；点击展开能看到完整结果或错误。
5. 不出现内部工具名、调用 ID、原始协议字段。
6. 任务运行详情与会话执行过程采用同一套动作摘要和展开卡片；一次批量工具调用必须展示为多条独立动作，结果与错误准确回填到各自动作。
