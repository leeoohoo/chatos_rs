# Chat OS 客户端本地项目全流程测试计划

## 1. 测试目标

使用最新 macOS 客户端，从空目录创建并完成一个真实的全栈个人博客项目，逐项验证：

- Electron 客户端外壳与本地 Core 能正常启动和恢复。
- 本地目录授权、项目创建、文件、Git、终端和模型能力可用。
- 第一次选择模型后即可正常发送消息，不需要重复选择。
- Plan 模式可以生成真实需求、技术文档和项目任务。
- 普通执行任务可以在本地项目内生成前端、后端、数据和测试代码。
- 纯规划任务不会强制启动沙箱，也不会修改项目文件。
- 成功、失败、重试、审批和重启恢复流程都有可见结果。
- 最终博客可以真实运行，并通过页面、API、持久化和响应式验收。

## 5. 本轮发现并已修复的问题

### FIX-01 本地默认模型指向旧模型 ID

- 现象：本地自动审批报 `model config has no local API key`。
- 修复：本地模型同步和运行时解析会自动选择同名称、同 provider、同 model 的最新有效配置，并修复四类默认模型设置。
- 影响链路：命令审批、记忆总结、环境初始化、项目管理任务。

### FIX-02 自动审批偶发只返回文字

- 现象：模型没有调用 `approval_decision`，第一次命令失败，重试后才成功。
- 修复：缺失审批工具调用时自动进行一次强制工具调用重试；第二次仍失败则保持默认拒绝。
- 安全边界：没有放宽审批策略或默认允许行为。

### FIX-03 聊天模型下拉重复展示旧配置

- 根因：同一供应商、模型和显示名称同时存在旧的无凭据配置与新的有效配置，聊天选择器只按 `enabled` 过滤。
- 修复：聊天选择器按模型身份合并配置，优先使用带凭据、可用且更新的配置；没有凭据或权威配置已禁用的模型不再进入发送下拉。
- 兼容：历史会话若仍保存旧模型 ID，会按同一模型身份自动迁移到新的有效 ID。

### FIX-04 规划回执暴露内部 ID 和原始枚举

- 现象：规划数据本身正确，但最终回复展示内部 ID、`technical_overview` 等实现细节，且模型可能在回忆 ID 时产生字符偏差。
- 修复：规划 Prompt 和云端/本地动态系统策略要求最终回复只展示用户可识别的名称、数量、依赖和下一步；内部标识确需展示时必须逐字复制已验证结果。
- 回归：重启到新 Core 后重新核对规划，最终回执只展示需求、文档、任务和依赖的可读中文摘要，未出现内部 ID、枚举、JSON 或工具名称。

### FIX-05 临时“只规划”约束污染长期需求和实施任务

- 现象：规划回合的“不要改文件、不要运行命令”被写进需求详情、技术文档和工程初始化任务，点击执行后 Task Runner 收到互相冲突的指令。
- 修复：规划 Prompt 明确禁止把当前回合临时操作约束写入长期项目产物；Task Runner 根据 `is_planning_task` 注入明确的规划/实施任务性质，实施任务会忽略历史规划回合中的临时禁写文字。

### FIX-06 复杂实施任务达到固定 20 轮上限且失败无回执

- 现象：工程初始化已经完成依赖安装、构建和测试自修，但在输出最终结果前达到固定 20 轮上限；失败状态只写入任务记录，会话中没有用户可见的失败消息。
- 修复：本地实施任务最大迭代提高到 80；成功结果为空时写入默认完成回执；失败、取消和内部异常都会在会话中追加清晰的中文回执，再更新任务状态。

### FIX-07 失败任务重试复用旧幂等键

- 现象：重试接口生成了新 Turn ID，但仍使用固定任务 ID 作为幂等键，存储层返回第一次失败的旧 Turn，导致重试立即报 `Local Task Runner turn already exists`。
- 修复：幂等键加入执行次数，同一任务的每次重试使用独立键，同时保留单次尝试内的幂等保护。

### FIX-08 本地 Task Runner 偏离全局 600 轮约定

- 现象：底层 AI Runtime 默认最大 600 轮，但本地 Task Runner 单独硬编码为 80，前端代码、测试和浏览器检查已完成后，因未及时输出最终总结而被错误标记失败。
- 修复：本地 Task Runner 不再维护独立数字，直接引用全局 Agent 的唯一默认值 600；第 599 轮起禁用工具并注入强制收口指令，要求根据真实操作立即输出总结，未完成项必须如实说明。
- 回归：针对性策略测试 2 passed；旧 80 轮失败任务使用独立重试 Turn 后成功完成并写回成功回执，后续任务已切换到 600 轮 Core。

### FIX-09 Electron 正常 IPC 关闭被误记为 Core 错误

- 现象：旧 Core 日志 60043 行中有 10175 条 `error shutting down connection`，每批正常短连接都被当作失败。
- 根因：Hyper 错误先被格式化成单一字符串，后续过滤器无法读取底层错误链。
- 修复：保留底层错误源，仅忽略确定性的正常连接关闭，其他 IPC 错误仍记录。
- 回归：新 App 连续 100 次 IPC 请求后该错误为 0；Core 全量 232 passed。

### FIX-10 Core 异常退出后 Electron 不会自动恢复

- 现象：终止 Core 后 Electron 只记录退出，5 秒后仍没有新 Core 进程。
- 修复：异常退出自动重启，采用 250ms 起步、最大 5 秒的指数退避；正常退出应用时禁用重启。
- 回归：真实终止 Core 后 250ms 拉起新 PID，健康接口恢复 200；旧设备租约过期后自动恢复云端连接。

### FIX-11 User Service 短暂中断会清空登录态与启动数据

- 现象：User Service 中断时刷新页面会永久退出登录，或把联系人/项目显示为空，并暴露内部服务地址与原始请求错误。
- 根因：Auth bootstrap 把所有异常都当成 token 失效；StrictMode 清理打断一次性启动链路；强制刷新复用旧并发请求；启动重试窗口过短。
- 修复：仅 401/403 清除登录；网络/5xx 保留本地会话；启动链按项目→联系人→会话执行并在 StrictMode 下完成；强制刷新覆盖旧请求；恢复重试窗口扩展到约 50 秒；用户错误统一脱敏。
- 回归：中断期间保留账号并只显示“服务暂时不可用”；服务恢复后联系人、原项目和 Plan 自动回填，无需手动刷新。
