# Chatos Internal Context Bilingualization Implementation Plan

## Goal

Add an English version for **system-internal runtime context only**, and let users switch the language from the existing runtime settings panel.

The switch should affect only context generated by Chatos itself, including:

1. base system fallback prompt selection behavior where applicable
2. builtin MCP prompt sections
3. task board prompt
4. contact / skill / command runtime prompts
5. runtime guidance formatting
6. internal summary-generation prompts and summary wrappers
7. any other system-generated prompt fragments injected by the backend

It should **not** translate or rewrite external inputs. These should remain as-is:

1. user messages
2. attachments
3. tool outputs
4. browser/web extracted content
5. remote command outputs
6. existing user-authored system contexts
7. plugin / skill / command full text loaded from external or user-managed sources unless we explicitly own both language variants
8. memory-engine-composed summary text returned from platform APIs, including the `merged_summary` blocks used in the main chat path

In short:

- internal Chatos-authored context: switchable `zh-CN` / `en-US`
- external or user-authored context: preserve original text and splice normally

## Current State

### 1. Runtime settings already provide a good configuration entry

The existing runtime settings modal is implemented in:

- `chat_app/src/components/UserSettingsPanel.tsx`

Current persisted keys are:

- `MAX_ITERATIONS`
- `LOG_LEVEL`
- `HISTORY_LIMIT`
- `CHAT_MAX_TOKENS`

Backend storage and merge path:

- `chat_app_server_rs/src/api/user_settings.rs`
- `chat_app_server_rs/src/services/user_settings.rs`

These settings are already loaded into chat runtime in:

- `chat_app_server_rs/src/api/chat_v3.rs`
- `chat_app_server_rs/src/api/chat_v2.rs`

and applied to AI clients through:

- `apply_settings_to_ai_client(...)`

This means the cleanest place to add the language selector is:

- new user setting key, for example `INTERNAL_CONTEXT_LOCALE`

### 2. Internal context is currently mostly Chinese hardcoded text

The major Chatos-owned prompt sources currently contain Chinese literals:

1. Builtin MCP prompt source
   - `BUILTIN_MCP_PROMPT.md`
2. Task board prompt
   - `chat_app_server_rs/src/services/task_board_prompt.rs`
3. Contact / command prompt builder
   - `chat_app_server_rs/src/core/chat_runtime_contact/prompt_builder.rs`
4. Summary prompt and wrapper for the local summarizer module
   - `chat_app_server_rs/src/services/summary/types.rs`
5. Some system-context assistant helper prompts
   - `chat_app_server_rs/src/services/system_context_ai/operations.rs`
6. A small amount of runtime-guidance wrapper text
   - `chat_app_server_rs/src/services/runtime_guidance_manager/support.rs`

### 3. Main chat context assembly path

Current v3 path:

1. frontend sends runtime options to `/agent_v3/chat/send|stream`
2. backend resolves model runtime
3. backend resolves chat stream context in `resolve_chat_stream_context(...)`
4. backend builds prefixed input items:
   - `contact_system_prompt`
   - `task_board_prompt`
   - `builtin_mcp_system_prompt`
   - `command_system_prompt`
5. AI client later adds:
   - runtime guidance items
   - merged summary from memory engine `compose_context(...)`, injected as returned
   - pending history
   - current user input

Relevant files:

- `chat_app_server_rs/src/api/chat_v3.rs`
- `chat_app_server_rs/src/api/chat_stream_common/runtime_context.rs`
- `chat_app_server_rs/src/services/task_board_prompt.rs`
- `chat_app_server_rs/src/services/v3/ai_client/stateless_context.rs`

## Scope Definition

### In scope

1. Add locale setting in runtime settings UI and backend persistence
2. Use locale during backend context composition
3. Provide English variants for Chatos-owned internal context fragments
4. Keep current Chinese behavior as default
5. Make v2 and v3 consistent
6. Preserve turn runtime snapshots and inspectors with the actual generated language

### Out of scope

1. Translating old stored summaries in database
2. Translating user-authored system context content
3. Translating tool results or external retrieved content
4. Full product UI i18n
5. Translating every document in `docs/plans`
6. Translating or rewriting memory-engine returned `merged_summary` text in the main chat path

## Recommended Design

### 1. Add a user setting key for internal context locale

Add:

- `INTERNAL_CONTEXT_LOCALE`

Recommended allowed values:

- `zh-CN`
- `en-US`

Default:

- `zh-CN`

Why this key:

1. it matches the existing runtime settings mechanism
2. it is user-scoped, not global
3. it can be loaded by both v2 and v3 without changing the request body contract
4. it is stable enough to later expand to more internal-language knobs if needed

### 2. Resolve locale once per chat request

Add a small resolver, for example:

- `chat_app_server_rs/src/core/internal_context_locale.rs`

Responsibility:

1. read `INTERNAL_CONTEXT_LOCALE` from effective user settings
2. normalize invalid values to default `zh-CN`
3. expose helpers such as:
   - `is_english()`
   - `is_chinese()`
   - `locale_tag()`

Recommended enum:

```rust
pub enum InternalContextLocale {
    ZhCn,
    EnUs,
}
```

This locale should be resolved in:

- `chat_app_server_rs/src/api/chat_v3.rs`
- `chat_app_server_rs/src/api/chat_v2.rs`

after `get_effective_user_settings(...)` and before prompt assembly / refresh.

### 3. Thread locale through all internal prompt builders

Refactor internal prompt composition functions to accept locale explicitly.

Recommended target surfaces:

1. `compose_builtin_mcp_system_prompt(...)`
2. `compose_effective_builtin_mcp_system_prompt(...)`
3. `build_task_board_prompt(...)`
4. `build_runtime_prefixed_messages(...)`
5. `build_runtime_prefixed_input_items(...)`
6. `compose_contact_system_prompt(...)`
7. `compose_contact_command_system_prompt(...)`
8. `format_runtime_guidance_instruction(...)`
9. summary prompt builders in `services/summary/types.rs`

Key rule:

- locale should be passed to internal prompt builders, not inferred deep inside from globals

This keeps behavior testable and avoids hidden coupling.

### 4. Split Chatos-owned text resources from composition logic

The biggest maintainability risk is duplicating English/Chinese strings inline across Rust functions.

Recommended structure:

1. keep composition logic in Rust
2. move large locale-dependent text blocks into dedicated resource helpers

Suggested modules:

- `chat_app_server_rs/src/i18n/internal_context/task_board.rs`
- `chat_app_server_rs/src/i18n/internal_context/contact_runtime.rs`
- `chat_app_server_rs/src/i18n/internal_context/runtime_guidance.rs`
- `chat_app_server_rs/src/i18n/internal_context/summary.rs`
- `chat_app_server_rs/src/i18n/internal_context/builtin_mcp.rs`

For builtin MCP prompt, prefer resource files:

- `BUILTIN_MCP_PROMPT.zh-CN.md`
- `BUILTIN_MCP_PROMPT.en-US.md`

and update the loader to select source by locale instead of a single hardcoded `include_str!(...BUILTIN_MCP_PROMPT.md)`.

This is better than embedding both languages into one markdown file because:

1. section order remains readable
2. translators can work on one file directly
3. later diffs stay manageable

### 5. Preserve external text as-is

Do not translate these during chat assembly:

1. `merged_summary` content returned by memory engine, including previously stored or newly composed platform summaries
2. historical messages loaded from session store
3. tool outputs and function call outputs
4. plugin / skill / command body content fetched from memory readers
5. active system context content created by users

This matches the intended rule:

- Chatos internal wrapper text changes language
- payload text from outside remains untouched

### 6. Treat summary generation separately

There are two different summary concerns:

1. the **summary-generation prompt**
2. the **summary content already persisted**

Recommended behavior:

1. new summary-generation prompts should respect locale
2. new summary wrapper text should respect locale
3. already-persisted historical summary text should not be translated on read
4. memory-engine returned `merged_summary` in the main chat path should continue to pass through unchanged

Implication:

- if a conversation was summarized in Chinese and the user later switches to English, the wrapper can be English, but the stored summary body may still be Chinese until new summarization happens

This is acceptable for first phase and avoids expensive migration work.

## Risks

### 1. Prompt drift between languages

If English and Chinese prompts diverge semantically, model behavior may change unexpectedly.

Mitigation:

1. maintain paired source files with same section structure
2. keep labels and rules semantically equivalent
3. add snapshot-style tests for key sections

### 2. Mixed-language context in existing sessions

After switching locale, old summaries and old user-authored contexts may remain Chinese.

Mitigation:

1. accept this in first phase
2. document that locale switch affects newly generated internal context only

### 3. Over-translation of external content

If we accidentally translate tool outputs or stored external text, we risk correctness loss.

Mitigation:

1. keep localization at wrapper-builder level only
2. never run translation on raw external payload text in the request path

### 4. Refresh-path inconsistency

Task board refresh and runtime guidance are injected in multiple paths.

Mitigation:

1. pass locale through refresh context store
2. audit both v2 and v3 paths before rollout

## Recommendation

Use a **user-scoped runtime setting** named `INTERNAL_CONTEXT_LOCALE`, default it to `zh-CN`, and localize only Chatos-owned internal prompt wrappers and prompt source files.

Do not attempt request-time translation of external content.

This gives you:

1. low-risk rollout
2. clean boundary between owned and non-owned context
3. compatibility with existing session data
4. minimal API contract churn

## Acceptance Criteria

This work is complete when:

1. users can switch internal context language in runtime settings
2. new chat turns use English internal context when `en-US` is selected
3. Chatos-owned prompt fragments are bilingual
4. external context remains untouched
5. v2 and v3 behave consistently
6. runtime snapshot and inspector show the real generated language for each turn
